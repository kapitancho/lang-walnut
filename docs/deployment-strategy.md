# Deployment Strategy

This document describes how Walnut is distributed and used across four deployment scenarios.

---

## Concepts

| Artifact | Description | Distribution |
|----------|-------------|--------------|
| `src/` | Almond engine PHP source | Part of `walnut/lang` package |
| `vendor/` | PHP dependencies | Generated by Composer |
| `bin/almond-*` | User-facing CLI scripts | Part of `walnut/lang` package |
| `almond/core-nut-lib/` | Walnut standard library (`.nut` files) | Bundled inside PHAR; shipped with Composer package |
| `almond/nutcfg.json` | Fallback config (points to bundled core-nut-lib) | Bundled inside PHAR |
| `walnut-src/` | **User's application code** | Never shipped; always project-local |
| `nutcfg.json` (project root) | Project config — created by `walnut init` | Written on init, owned by user |

**Resolution order for `nutcfg.json`:**
1. `<cwd>/nutcfg.json` — user's project config (highest priority)
2. `<engine-root>/almond/nutcfg.json` — bundled fallback (used when running outside a project)

---

## Developer Workflow (all scenarios)

The workflow is identical regardless of how walnut was installed:

```bash
# 1. Create project
mkdir my-project && cd my-project
walnut init

# 2. Write code in walnut-src/

# 3. Use the tools
walnut cli main arg1 arg2     # run a module
walnut test                   # run .test.nut files
walnut analyse                # type-check all modules
walnut serve                  # start HTTP server
walnut lsp                    # start LSP server (used by VS Code extension)
```

`walnut init` sets up the project scaffold:
```
my-project/
  nutcfg.json        # written by init; points to extracted core-nut-lib and walnut-src
  core-nut-lib/      # extracted from inside walnut.phar (or from vendor/ in Composer scenario)
  walnut-src/        # created empty; user adds .nut files here
```

The `nutcfg.json` written by `init`:
```json
{
  "sourceRoot": "./walnut-src",
  "packages": {
    "core": "./core-nut-lib"
  }
}
```

## Scenario 1 — Clone the Repository Directly

**Use case:** Contributors, language developers, early adopters who want the full source.

```bash
git clone https://github.com/walnut-lang/walnut
cd walnut
composer install
```

Scripts are run directly from `bin/`:

```bash
./bin/almond-cli <module> [args]
./bin/almond-test [folder]
./bin/almond-analyse [folder]
./bin/almond-http          # open work — does not exist yet
```

The `almond/nutcfg.json` fallback is picked up automatically when running from within the repo,
pointing at `almond/core-nut-lib` and `almond/walnut-src` for demos and internal testing.

**Open work:**
- `bin/almond-http` does not exist yet — needs to be created.
- `bin/almond-init` does not exist yet — lower priority here since contributors use the repo directly.

## Scenario 2 — Include via Composer (`composer require`)

**Use case:** PHP projects that embed Walnut as a library or scripting tool.

```bash
composer require walnut/lang
```

Composer exposes scripts via `vendor/bin/`:

```bash
vendor/bin/almond-cli <module> [args]
vendor/bin/almond-test [folder]
vendor/bin/almond-analyse [folder]
vendor/bin/almond-http
vendor/bin/almond-init       # sets up nutcfg.json + walnut-src/ in the project
```

`almond-init` in this scenario writes a `nutcfg.json` pointing to the Composer-installed library:

```json
{
  "sourceRoot": "./walnut-src",
  "packages": {
    "core": "./vendor/walnut/lang/almond/core-nut-lib"
  }
}
```

**nutcfg resolution:** `<cwd>/nutcfg.json` takes priority. The bundled fallback at
`vendor/walnut/lang/almond/nutcfg.json` is never used once the user has run `almond-init`.

**Open work:**
- `composer.json` `"bin"` entries have been updated to `almond-cli`, `almond-test`, `almond-analyse`.
  `almond-http` and `almond-init` will be added once the scripts are created.
- `bin/almond-http` needs to be created.
- `bin/almond-init` needs to be created (replaces the old `walnut-init-project` which referenced
  the removed `empty-src/` directory and the old `core-nut-lib` path).

## Scenario 3 — PHP Installed, Download the PHAR

**Use case:** Users who have PHP but do not use Composer. Single-file install.

```bash
# Install (once)
curl -L https://github.com/walnut-lang/walnut/releases/latest/download/walnut.phar \
  -o /usr/local/bin/walnut
chmod +x /usr/local/bin/walnut

# Create project (once per project)
mkdir my-project && cd my-project
walnut init

# Daily use
walnut cli main arg1
walnut test
walnut analyse
walnut serve
```

**What lives inside `walnut.phar`:**

| Path inside PHAR | Contents |
|------------------|----------|
| `src/` | Almond engine PHP source |
| `vendor/` | PHP dependencies |
| `bin/almond-*.php` | Entry point scripts |
| `almond/core-nut-lib/` | Walnut standard library |
| `almond/nutcfg.json` | Bundled fallback config |

`core-nut-lib` is **inside** the PHAR so `walnut init` can extract it to the project directory.
There is no separate `core-nut-lib` download — the PHAR is the only file to download.

**Stub routing** (`phar/stub.php`):

| Command | Entry point inside PHAR |
|---------|-------------------------|
| `walnut init` | `bin/almond-init.php` — extracts `core-nut-lib`, writes `nutcfg.json` |
| `walnut cli <module>` | `bin/almond-cli.php` |
| `walnut test [folder]` | `bin/almond-test.php` |
| `walnut analyse [folder]` | `bin/almond-analyse.php` |
| `walnut serve` | `bin/almond-http.php` |
| `walnut lsp` | `bin/almond-lsp.php` — starts the LSP server for VS Code |
| `walnut --version` | stub (no engine needed) |

**LSP + VS Code integration:**

The VS Code extension configures `walnut lsp` as its server command. This means:
- No separate LSP download for PHAR users.
- The extension only needs to know the path to `walnut` (or `walnut.phar`).
- The `walnut.server.path` VS Code setting points to the PHAR.

All Scenario 3 open items resolved: `phar/stub.php` updated with full subcommand routing;
`box.json` updated to bundle `almond/` (core-nut-lib + nutcfg.json), `lsp/server/src/`, and `http/`;
`bin/almond-init.php`, `bin/almond-http.php`, and `bin/almond-lsp.php` created.

## Scenario 4 — No PHP (Future)

**Use case:** End-users who cannot or should not install PHP. Single native binary per platform.

The workflow is identical to Scenario 3 — the binary replaces `php walnut.phar`:

```bash
walnut init
walnut cli main
walnut lsp      # VS Code uses this
```

**Options (in preference order):**

### 4a. FrankenPHP Static Build (preferred)
- Compiles PHP + the PHAR into a single static binary per platform.
- Ships as `walnut` (Linux x64/arm64), `walnut.exe` (Windows), `walnut-darwin` (macOS arm64/x64).
- No PHP installation required on the user's machine.
- The release workflow would add a matrix over platforms and compile via FrankenPHP.

### 4b. RoadRunner Bundle (fallback, HTTP only)
- Pre-built `rr` binary + PHAR. Only meaningful for the HTTP server use case — not CLI or LSP.
- More complex setup; not recommended as the primary path.

### Architecture constraint
No engine changes are needed for Scenario 4 — it is purely a packaging concern.
The stub, entry point scripts, and `nutcfg.json` resolution all work identically inside a
FrankenPHP binary as they do inside a PHAR.

**When to implement:** After Scenario 3 is fully working and stable.
